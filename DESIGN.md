## Context

`yc_roulette_prototype.html` is a **UI/UX mockup only** — it defines the visual design
system, screen layouts, and interaction patterns, but contains no game logic. The actual
game (session construction, scoring, reveal sequence, share card, state machine) needs
to be built from scratch.

`yc_dataset.js` is production-ready: 215 curated YC companies with all required fields
(`name`, `batch`, `blurb`, `industry`, `lastActiveYear`, `funding`).

The PRD (`yc_roulette_prd.docx`) is the authoritative requirements spec.

## Problem Statement

Build a fully functional YC Roulette game — a mobile-first estimation game where players
read a real YC startup brief and guess three things: last active year, total funding
raised, and industry. 5 startups per round, closer guesses score more points, shareable
score card at the end. Target audience: startup/tech Twitter crowd.

The prototype gives us the UI skin. Everything underneath needs to be built.

## What Needs to Be Built

### Game state machine
- Session state: current startup index (0–4), cumulative score, per-startup results
- Screen transitions: home → startup card → guesses → reveal → (next or final card)
- No back-navigation during a round

### Session construction (per PRD §6.3)
- Draw 5 startups from 215, without replacement
- Seed with 1–2 famous anchors per session (Airbnb, Stripe, Dropbox, DoorDash, Reddit,
  Coinbase, Twitch, Instacart, GitLab, Gusto)
- Target mix: 2–3 alive (lastActiveYear=2025), 2–3 dead/exited
- Industry variety: no more than 2 startups from the same industry per session
- Shuffle remaining 3–4 slots from non-anchor pool with above constraints
- Retry cap: 100 attempts. If constraints cannot be satisfied, return best attempt
  assembled so far (industry variety is the first constraint to relax). Game always plays.

### Three guess mechanics (per PRD §4.3)

**Guess 1 — Last Active Year**
- Horizontal slider from the company's founding year (derivable from batch code) to 2025
- Year label updates live on every drag event
- Founding year derived from batch last two digits: S05→2005, W23→2023, S02→2002
  Formula: `2000 + parseInt(batch.slice(1), 10)`
- Slider default position: midpoint of [foundingYear, 2025] — i.e.,
  `Math.round((foundingYear + 2025) / 2)`. Neutral; doesn't hint at the answer.

**Guess 2 — Total Funding Raised**
- 4 tappable chip options: `seed` / `<5m` / `5-50m` / `50m+`
- Single-select, tap to toggle
- Tier ordering (low to high): seed → <5m → 5-50m → 50m+
  Off-by-one (5pts) applies to adjacent tiers in this sequence only.

**Guess 3 — Industry**
- 8 tappable chip options: Fintech / Healthtech / Dev Tools / E-commerce /
  Enterprise SaaS / Consumer / Logistics / AI/ML
- Single-select

Submit button: disabled until all 3 guesses are made.

### Scoring system (per PRD §4.4)

Max 30 points per startup, 150 per game. Minimum 5 per game (year miss floor of 1pt × 5).

| Condition | Points |
|---|---|
| Last active year — exact match | 10 |
| Last active year — within 1 year | 8 |
| Last active year — within 3 years | 5 |
| Last active year — off by 4+ years | 1 |
| Funding bucket — exact match | 10 |
| Funding bucket — off by one tier | 5 |
| Industry — exact match | 10 |
| Industry — adjacent category | 3 |

Industry adjacency map (fixed, maintained in game logic):
- Fintech ↔ Enterprise SaaS
- Dev Tools ↔ AI/ML
- E-commerce ↔ Consumer
- Logistics ↔ E-commerce
- Healthtech ↔ Enterprise SaaS
- Other: no adjacencies — exact match only (10pts), miss = 0pts

Note: `yc_dataset.js` uses `"AI / ML"` (with spaces). This must be normalized to `"AI/ML"`
in the dataset before shipping. 10 companies use industry `"Other"` — these are valid
entries and score normally.

### Reveal sequence (per PRD §7.2)
- Reveal year answer first, then funding, then industry — 300ms stagger
- Each reveal: card flip animation, points earned pop (counts up from 0, 400ms ease-out)
- Green for exact, amber for partial, red for miss
- Round subtotal at bottom after all 3 revealed
- Tap to continue to next startup (or final card after startup 5)

### Final score card (per PRD §4.5 + §5)
Score tiers:
- **S tier** 135–150 pts — "YC partner energy"
- **A tier** 105–134 pts — "TechCrunch power reader"
- **B tier** 75–104 pts — "VC dinner survivor"
- **C tier** <75 pts — "Startup world tourist"

One-liner bank: 5 per tier, one randomly selected per game completion.
No dedup tracking needed — Play Again resets all state, so each game gets a fresh random
pick. See PRD §5.2 for full one-liner list.

Card contents: game logo, score (e.g. 112/150), tier badge, one-liner, 5-startup
breakdown (correct/partial/wrong per guess), Save as Image + Share buttons, Play Again CTA.

### Share card (per PRD §5.3)
- Render the score card in an off-screen DOM node for clean html2canvas capture
- Save as image: html2canvas → PNG → Blob URL → browser download
- Canvas validation: sample 100 random pixels; if >90% are #ffffff or rgba(0,0,0,0),
  treat as blank — show "Screenshot this to save your score" toast instead of blank PNG
- Native share: navigator.share() with pre-filled text + URL
- Desktop fallback: clipboard copy + "Copied to clipboard — paste to share" toast
- navigator.share() detection: `if (navigator.share)` before calling

### Persistence
- localStorage key: `yc_roulette_high_score` (integer)
- Update after each game if new score beats stored high score
- Display on home screen if localStorage value exists
- Wrap all localStorage calls in try/catch — on failure (private browsing, quota), silently skip

## Constraints

- Vanilla HTML/CSS/JS — no framework, no build step, matches prototype's tech approach
- Three co-located files: `index.html` + `game.js` + `yc_dataset.js` — no CDN dependencies at runtime
- Google Fonts loaded via `<link>` (Nunito + DM Serif Display) — acceptable network dependency
- html2canvas loaded via CDN `<script>` tag — only needed for share card, lazy-ok
  Inject on first visit to reveal screen (not on page load) to meet <1.5s FCP.
  Add `onerror` handler: if script fails to load, catch and show
  "Screenshot this to save your score" toast when Save is tapped.
- No backend, no auth, no analytics for V1
- Mobile-first: all interactions thumb-reachable on 375px–430px viewport
- Animations <400ms (per PRD §7.1)
- First Contentful Paint <1.5s on mid-range Android on 4G (per PRD §8.2)

## Design System (from prototype)

Carry forward exactly from `yc_roulette_prototype.html`:
- CSS custom properties (`:root` variables): `--cream`, `--orange`, `--green`, `--sand`, `--ink`, `--muted`, `--radius`
- Font stack: `'Nunito'` (body/UI) + `'DM Serif Display'` (headings/company names)
- Phone shell: 360×740px, `border-radius: 44px`, centered on `#F0E8DC` canvas
- All component styles: buttons, chips, slider, cards, animations

The prototype CSS is the source of truth for visual design. Do not redesign — implement.

## Approach

**Build on the prototype HTML as the shell.**

The prototype already has all screen layouts, CSS, and animation keyframes defined.
The implementation work is:
1. Add the JS game engine beneath the existing HTML/CSS structure
2. Wire screen transitions to the state machine
3. Implement the three guess inputs (slider + chips) with live state
4. Implement scoring, reveal sequence, and final card logic
5. Implement share card capture and distribution

This is faster than a rewrite and guarantees pixel-perfect fidelity to the approved
UI/UX design.

**Do not port to React.** The PRD lists React as optional. For a single-session game
with no server state, vanilla JS is simpler to deploy and debug. A React rewrite adds
a build step and a bundler for no user-facing benefit.

## Screen-by-Screen Build Spec

### Screen 1: Home
- Show "YC Roulette" wordmark + tagline
- If `localStorage.yc_roulette_high_score` exists: show "Your best: X / 150 pts"
- Single CTA: "Start playing →"
- On tap: run session construction, transition to Screen 2

### Screen 2: Startup Card
- Display: company name (DM Serif Display), blurb, YC batch badge
- No industry/funding/outcome hints shown
- CTA: "Make your guesses →"
- On tap: transition to Screen 3

### Screen 3: Guesses
- Year slider (founding year → 2025, live label update)
- Funding chips (4 options, single-select)
- Industry chips (8 options, single-select)
- "Submit Guesses" button — disabled until all 3 selected
- On submit: run scoring for this startup, transition to Screen 4

### Screen 4: Reveal
- Reveal year answer → points pop (300ms)
- Reveal funding answer → points pop (300ms stagger)
- Reveal industry answer → points pop (300ms stagger)
- Show round subtotal
- "Continue →" — if startup index < 4: transition to Screen 2 (next startup)
               — if startup index = 4: transition to Screen 5

### Screen 5: Final Score Card
- Compute tier from total score
- Select random one-liner from tier's bank of 5
- Render off-screen card for html2canvas
- Show Save as Image + Share buttons
- "Play Again" → reset state, regenerate session, go to Screen 1

## Success Criteria

### Functional
- Full 5-startup round completes without errors in Chrome, Firefox, Safari desktop
- All scoring rules from PRD §4.4 produce correct point values (unit-test mentally
  or via console during QA)
- Industry adjacency map matches PRD spec exactly
- Session construction: 1–2 anchors per session, ≤2 same-industry startups, mix of
  alive/dead — verify by logging session arrays to console during QA
- Submit button disabled until all 3 guesses made
- Reveal sequence fires in correct order with correct colors (green/amber/red)

### Share card
- html2canvas renders non-blank on iOS Safari 15+ AND Android Chrome 100+
- Blank-check (100px sample, >90% white = blank) triggers toast fallback
- navigator.share() opens OS share sheet on mobile
- Clipboard fallback works on desktop, shows "Copied!" toast

### Performance
- Page loads and first card renders in <1.5s on throttled 4G (Chrome DevTools)
- All animations complete in <400ms

### Persistence
- High score updates after each game if beaten
- High score survives tab close and reopen
- localStorage failure (private browsing) is silent — no error shown

### Device matrix
- Mobile (primary): iOS Safari 15+, Android Chrome 100+
- Desktop (secondary): Chrome 110+, Firefox 115+, Safari 16+
